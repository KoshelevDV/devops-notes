# BGP Monitoring через BMP (BGP Monitoring Protocol)

> Источник: [Как в Авито построили систему мониторинга BGP](https://habr.com/ru/companies/avito/articles/1012780/)
> Лаба: https://github.com/nocdeveloper/BMP-collector-demo

---

## Зачем мониторить BGP

- Видеть BGP flapping, route hijacking, аномалии апстримов **до** жалоб пользователей
- SNMP не даёт историю изменений маршрутов — только текущее состояние
- Syslog даёт текстовые события, но не структурированные данные по RIB

---

## Протокол BMP — RFC 7854

- Специализированный протокол для BGP-мониторинга
- Роутер **сам пушит** данные на коллектор (пассивный, не надо поллить)
- Отдаёт полную таблицу маршрутов: **Loc-RIB**, **Adj-RIB-In**, **Adj-RIB-Out**
- Типы сообщений: `Route Monitoring`, `Statistics Report`, `Peer Up/Down`

---

## Архитектура (Авито)

```
Роутеры/свитчи
     │ BMP (TCP push)
     ▼
OpenBMP Collector   ← принимает BMP-стримы от всех устройств
     │
     ▼
Kafka               ← буфер, decoupling, replay
     │
     ▼
ClickHouse          ← хранилище маршрутной истории
     │
     ▼
Grafana             ← дашборды и алерты
```

---

## Что мониторить

| Метрика | Описание | Аномалия |
|---|---|---|
| `total_changes` | Суммарные изменения в таблице маршрутизации | Резкий спайк |
| `total_withdraws` | Отозванные маршруты | Красный спайк = что-то упало |
| `total_updates` | Новые/обновлённые маршруты | Нетипичный burst |
| `peer_up/down` | Состояние BGP-сессий с пирами | down = потеря связности |

---

## Сценарии использования

- **BGP flapping** — сессия поднимается/падает циклично → нестабильный апстрим
- **Route hijacking** — чужой AS анонсирует твои префиксы → видно по смене AS_PATH
- **Утечка маршрутов (route leak)** — транзитный AS неожиданно анонсирует чужие маршруты
- **Плановые работы** — baseline изменений в таблице при обслуживании

## Лаба на docker-compose

Стек: OpenBMP + Kafka + ClickHouse + Grafana

```bash
git clone https://github.com/nocdeveloper/BMP-collector-demo
cd BMP-collector-demo
docker compose up -d
```

Позволяет потрогать систему без живого железа — симулятор генерирует BGP-события.

---

## Подводные камни

- ClickHouse требует ресурсов при большом объёме маршрутных таблиц (full DFZ = ~900k префиксов)
- OpenBMP надо слушать на отдельном IP/порту — роутеры конфигурируются явно (`bmp-server <ip> port <port>`)
- Kafka важен как буфер: без него при перезапуске коллектора теряются события
- Не все вендоры одинаково реализуют BMP — проверяй поддержку Adj-RIB-Out (Cisco IOS XR, Juniper поддерживают, некоторые — только Loc-RIB)

## Альтернативы

| Решение | Особенность |
|---|---|
| **GoBMP** | Лёгкий Go-коллектор, проще OpenBMP |
| **pmacct** | Универсальный коллектор (NetFlow/IPFIX/BMP/BGP), сложнее |
| **RTBH + FlowSpec** | Реактивное управление маршрутами (блокировка DDoS), не мониторинг |
| **Kentik / ThousandEyes** | SaaS BGP monitoring, дорого |
