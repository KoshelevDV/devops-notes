# Автоматизация ревью документации через AI

## Идея

Рутинную проверку однотипных документов можно делегировать LLM-агенту.
Типичный кейс: агент встраивается в трекер (Jira), сам ходит в базу знаний (Confluence/GitLab Wiki),
прогоняет документ через LLM и выдаёт детальный отчёт за ~26 секунд вместо ручной проверки.

## Что проверяет агент

- Соответствие корпоративному шаблону / стандарту
- Наличие обязательных разделов
- Качество описания (полнота, однозначность)
- Орфография, терминология
- Устаревшие ссылки / версии

## Архитектура (self-hosted вариант)

```
Триггер (Jira webhook / GitLab MR / ручной запуск)
    │
    ▼
Агент (Python / n8n / OpenClaw)
    ├─ забирает документ из Confluence / GitLab Wiki / Git
    ├─ загружает корпоративный стандарт (шаблон, чеклист)
    └─ отправляет в LLM с промптом ревью
            │
            ▼
        LLM (GLM / GPT)
            │
            ▼
    Структурированный отчёт
        ├─ список нарушений с объяснением
        ├─ оценка по критериям
        └─ предложения по правкам
            │
            ▼
    Публикация результата
        ├─ комментарий в Jira / GitLab MR
        └─ уведомление в Telegram / Slack
```

## Реализация на нашем стеке

### Вариант 1 — GitLab MR pipeline

```yaml
# .gitlab-ci.yml
doc-review:
  stage: review
  rules:
    - changes:
        - "docs/**/*.md"
  script:
    - |
      for f in $(git diff --name-only HEAD~1 HEAD -- 'docs/*.md'); do
        curl -s -X POST http://10.0.30.18:8080/v1/chat/completions \
          -H "Content-Type: application/json" \
          -d "{
            \"model\": \"GLM-4.7-Flash\",
            \"messages\": [
              {\"role\": \"system\", \"content\": \"$(cat docs/.review-prompt.md)\"},
              {\"role\": \"user\", \"content\": \"$(cat $f)\"}
            ],
            \"max_tokens\": 2048
          }" | jq -r '.choices[0].message.content' > review_${f//\//_}.md
      done
  artifacts:
    paths:
      - review_*.md
```

### Вариант 2 — OpenClaw агент по триггеру

OpenClaw уже умеет:
- читать файлы из Git репо
- гнать через GLM (порт 8080)
- постить результат в Telegram / GitLab MR comment

Достаточно написать skill или cron-задачу на новые/изменённые `.md` файлы.

### Вариант 3 — n8n workflow (AF_Automation)

```
GitLab webhook (push) → n8n
    → HTTP: забрать файл через GitLab API
    → HTTP: отправить в GLM
    → HTTP: создать комментарий в MR через GitLab API
    → Telegram: уведомление
```

## Промпт для ревью (шаблон)

```
Ты — технический редактор. Проверь документ по следующим критериям:

1. Наличие обязательных разделов: [Overview, Architecture, Setup, API Reference, Troubleshooting]
2. Каждый раздел должен содержать минимум 3 предложения
3. Все команды должны быть в code block
4. Ссылки должны быть рабочими (проверь форматирование)
5. Нет устаревших версий инструментов (сверяй с контекстом)

Выведи:
- [ ] список нарушений с указанием раздела
- Оценку качества от 1 до 10
- Конкретные предложения по правкам (не переписывай сам документ)
```

## Что важно для качества ревью

- **Контекст стандарта** — LLM должен получать шаблон/чеклист вместе с документом, не только документ
- **Структурированный вывод** — просить JSON или чёткий формат, чтобы парсить программно
- **Дифф, не весь файл** — при MR достаточно гнать только изменённые части (экономия токенов)
- **Итеративность** — после правок автор может запросить повторное ревью той же командой
